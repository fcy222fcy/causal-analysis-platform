# 智慧养殖环境异常溯源与因果分析系统 — 答辩问题回答

---

## 1. 为什么这样设计数据库？

本系统共设计了 **9 张核心表**，采用以 `barn_id`（棚舍ID）为核心关联维度的星型模型变体：

| 表名 | 职责 | 设计理由 |
|---|---|---|
| `sys_user` | 用户与权限管理 | 支持 admin/user 角色，通过 `barn_id` 绑定管辖范围 |
| `env_sensor_data` | 环境传感器原始数据 | 高频写入（每3秒一条），按 `(barn_id, timestamp)` 建联合索引 |
| `animal_behavior` | 动物行为识别数据 | 与传感器数据分离，因数据来源和更新频率不同 |
| `anomaly_event` | 异常事件记录 | 从原始数据中"提炼"出的事件，单独存储避免污染原始数据流 |
| `causal_relation` | 因果关系分析结果 | 存储分析结果，`cause_variable` / `effect_variable` 支持变量间的因果图谱 |
| `traceability_report` | 溯源报告 | 以事件为粒度，存储因果链和报告内容 |
| `alarm_record` | 告警记录 | 有独立生命周期（pending → 已处理），关联但独立于异常事件 |
| `operation_log` | 操作日志 | 审计追踪，满足可追溯性需求 |
| `system_config` | 阈值配置 | 将阈值从硬编码改为数据库可配置，运维可动态调整 |

### 核心设计思路

**① 关注点分离 —— 数据处理流水线**

```
原始数据 (env_sensor_data)
    ↓ 异常检测
异常事件 (anomaly_event)
    ↓ 因果分析
因果关系 (causal_relation)
    ↓ 告警触发
告警记录 (alarm_record)
    ↓ 溯源报告
溯源报告 (traceability_report)
```

每一层只关心自己的职责，原始数据与分析结果解耦，便于独立扩展和维护。

**② 索引策略**

所有高频查询表都以 `(barn_id, timestamp)` 作为联合索引，覆盖"按棚舍 + 时间范围查询"的主要场景：

```sql
CREATE INDEX idx_sensor_barn_time ON env_sensor_data(barn_id, timestamp);
CREATE INDEX idx_behavior_barn_time ON animal_behavior(barn_id, timestamp);
CREATE INDEX idx_anomaly_barn_time ON anomaly_event(barn_id, timestamp);
```

**③ 时空局部性**

养殖场景下数据天然按"棚舍"分区，`barn_id` 作为逻辑分区键，避免跨区查询。系统模拟 5 个棚舍（`barnId` 1~5），每个棚舍独立采集和分析。

**④ 配置与日志分离**

阈值配置（`system_config`）和操作日志（`operation_log`）作为支撑表，使核心业务表保持简洁，同时满足运维和审计需求。

## 2. 核心算法怎么实现？

本系统有 **两个核心算法模块**：

### （1）异常检测算法（`AnomalyDetectionService`）

采用 **双层检测策略**：阈值检测 + 滑动窗口统计检测。

#### 第一层：阈值检测 —— 捕捉突变异常

```java
// 预设阈值
TEMP_THRESHOLD_HIGH = 35.00°C
AMMONIA_THRESHOLD_HIGH = 25.00ppm

// 检测逻辑
if (temperature > 35°C)  → 标记高温异常
if (ammoniaLevel > 25ppm) → 标记氨气超标
```

- 计算复杂度：O(n)，遍历一次数据即可
- 优点：简单快速，适合捕捉突发性异常
- 阈值可通过 `system_config` 表动态调整，无需重启

#### 第二层：滑动窗口统计检测 —— 捕捉渐变异常

```java
窗口大小 = 60 个数据点
异常阈值 = 2 倍标准差

for (每个数据点 i) {
    window = 第 i-60 到 i-1 个数据点
    mean = 窗口内均值
    std  = 窗口内标准差
    if (|当前值 - mean| > 2 × std) {
        → 标记为统计异常
    }
}
```

- 计算复杂度：O(n × w)，w 为窗口大小（60）
- 基于统计学的 3σ 原则（此处取 2σ 更敏感）
- 对正常波动具有鲁棒性，适合捕捉氨气浓度缓慢升高等渐变异常
- 温度和氨气分别独立做滑动窗口检测

#### 双层互补

| 检测方式 | 适用场景 | 举例 |
|---|---|---|
| 阈值检测 | 突变异常 | 传感器故障导致温度骤升到 40°C |
| 滑动窗口 | 渐变异常 | 通风系统故障导致氨气浓度在 30 分钟内从 15ppm 缓慢升到 28ppm |

### （2）因果分析（Java + Python 微服务）

#### 整体架构

```
Java 后端 (Spring Boot, port 8080)
  │  RestTemplate 调用
  ▼
Python 因果分析服务 (FastAPI, port 5000)
  │  DoWhy 因果推断
  ▼
返回因果图 + 相关性矩阵 + 因果路径
  │
  ▼
Java 后端缓存到 Redis（10分钟 TTL）
```

#### Python 端核心逻辑（`causal_analyzer.py`）

**① 相关性分析**

计算温度、湿度、氨气浓度三个变量之间的 Pearson、Spearman、Kendall 相关矩阵。

**② DoWhy 因果推断（主算法）**

```python
# 1. 构建因果 DAG（领域知识）
dag = {
    "temperature → humidity",      # 温度影响湿度
    "ammonia → temperature",       # 氨气浓度影响温度
    "ammonia → humidity"           # 氨气浓度影响湿度
}

# 2. 使用 DoWhy 的 backdoor.linear_regression 方法
causal_estimate = causal_model.estimate_effect(
    method_name="backdoor.linear_regression"
)

# 3. 输出因果强度和方向
```

**③ 降级策略**

当 DoWhy 不可用或执行失败时，自动降级为相关性分析作为因果强度的近似。

#### Java 端异步任务管理

```java
// 线程池配置
corePoolSize = 5, maxPoolSize = 10, queueCapacity = 100
rejectedExecutionHandler = CallerRunsPolicy  // 队列满时由调用线程执行，不丢弃任务

// 异步执行流程
CompletableFuture.runAsync(() -> {
    1. 调用 Python 服务
    2. 缓存结果到 Redis（10分钟过期）
}, taskExecutor);
```

使用 `ConcurrentHashMap` 存储任务状态（PENDING → RUNNING → COMPLETED/FAILED），支持前端轮询任务进度。

## 3. 系统如何扩展？

### 3.1 水平扩展

| 扩展方向 | 具体方案 |
|---|---|
| **微服务拆分** | 当前单体架构可按模块拆分为：传感器服务、分析服务、告警服务，通过 API 网关统一入口 |
| **Python 服务独立扩容** | 因果分析服务已经是独立微服务，可通过容器化（Docker + K8s）水平扩展 |
| **消息队列解耦** | 传感器数据写入可用 Kafka / RabbitMQ 缓冲，解耦数据采集与处理 |
| **前端 CDN** | Vue 构建产物可部署到 CDN，支持静态资源加速 |

### 3.2 垂直扩展

| 扩展方向 | 具体方案 |
|---|---|
| **新增传感器类型** | `env_sensor_data` 表通过添加字段或新建传感器数据表扩展（如光照、噪声） |
| **新增棚舍** | `barn_id` 字段天然支持，无需修改表结构 |
| **新增分析算法** | 在 `AnomalyDetectionService` 中添加新的检测策略（如 LSTM、Isolation Forest），保持接口不变 |
| **新增告警渠道** | 当前已有 WebSocket 推送，可扩展邮件、短信、钉钉/企微通知 |
| **新增因果变量** | Python 端 DAG 可动态添加新节点和边，支持更多环境因子的因果分析 |

### 3.3 配置化扩展

- **阈值可配置**：`system_config` 表存储阈值，运维可在不重启的情况下调整检测参数
- **定时任务可调**：`@Scheduled(fixedRate=3000)` 等参数可通过 Spring Cloud Config 或 Nacos 动态修改
- **Python 服务地址**：通过 `python.causal.service.url` 配置，支持切换不同环境

### 3.4 代码层面扩展性

```
Controller → Service → Mapper（MyBatis-Plus）
```

- MyBatis-Plus 提供 CRUD 基础，新增表只需添加 Entity + Mapper
- 分层架构职责清晰，新增功能按层次添加即可
- `@Async` + `CompletableFuture` 异步任务框架可复用于其他耗时操作
- DTO / VO 分离，前后端数据模型独立演进

## 4. 如何处理大数据量？

本系统从 **采集、存储、查询、计算** 四个层面应对大数据量：

### 4.1 采集层

| 策略 | 实现 |
|---|---|
| **定时任务控频** | `@Scheduled(fixedRate=3000)` 限制写入频率，避免瞬时压力 |
| **批量写入支持** | `batchSaveSensorData` 支持批量插入，生产环境可改为 MyBatis-Plus 的 `saveBatch` |
| **数据模拟** | 启动时批量生成 200 条传感器 + 100 条行为历史数据，后续定时增量写入 |

### 4.2 存储层

| 策略 | 实现 |
|---|---|
| **多环境数据源** | 开发环境 H2 内存数据库（零运维）；生产环境 MySQL（支持分区表和分库分表） |
| **数据归档** | 可通过定时任务将历史数据迁移到归档表或冷存储 |
| **逻辑删除** | MyBatis-Plus 配置 `deleted` 字段，软删除避免数据丢失 |

### 4.3 查询层

| 策略 | 实现 |
|---|---|
| **联合索引** | `(barn_id, timestamp)` 联合索引覆盖最常见查询路径，避免全表扫描 |
| **分页查询** | `PageResultVO` 封装分页结果，前端分页展示 |
| **LIMIT 限制** | 异常检测查询 `LIMIT 200`，避免一次加载过多数据 |
| **条件构造器** | MyBatis-Plus 的 `LambdaQueryWrapper` 自动防止 SQL 注入 |
| **时间范围查询** | `getSensorDataByTimeRange` 支持按时间区间精确查询 |

### 4.4 计算层

| 策略 | 实现 |
|---|---|
| **Redis 缓存** | 因果分析结果缓存 10 分钟，避免重复调用 Python 服务 |
| **线程池隔离** | 独立线程池处理分析任务，不影响主业务线程 |
| **滑动窗口流式** | 异常检测只取最近 200 条数据计算，时间复杂度 O(n×w) |
| **异步任务管理** | `ConcurrentHashMap` 存储任务状态，支持异步查询进度 |
| **优雅降级** | Redis 不可用时跳过缓存直接查库；Python 服务不可用时返回降级因果图 |

### 4.5 未来优化方向

```
┌─────────────────────────────────────────────────────┐
│  当前架构                     优化方向                │
├─────────────────────────────────────────────────────┤
│  MySQL 单库           →  时序数据库 (InfluxDB/TDengine)│
│                          天然支持时间序列查询和降采样    │
│                                                     │
│  直接写入数据库        →  消息队列削峰 (Kafka)         │
│                          异步消费处理，解耦采集与存储    │
│                                                     │
│  单节点 MySQL         →  读写分离 (主从复制)           │
│                          读操作走从库                  │
│                                                     │
│  全量数据存储          →  数据分区 (按月分区)           │
│                          查询和归档更高效              │
│                                                     │
│  同步异常检测          →  Flink/Spark Streaming       │
│                          实时流式计算，毫秒级响应       │
└─────────────────────────────────────────────────────┘
```
